# Proxy Provider Evaluation Guide

当用户询问 "这个节点靠不靠谱"、"跑路概率大不大" 或要求推荐节点服务时使用。

## 核心原则

**不要推荐具体的商家。** 节点服务质量高度动态、因人因地而异，且推荐本身有合规风险。可以提供评估框架，让用户自行判断。

## 跑路信号 Checklist

向用户展示以下信号，帮助他们自行评估：

| 信号 | 危险程度 | 说明 |
|------|---------|------|
| 强制年付/长期套餐 | ⚠️ 高 | 稳定服务敢做月付，上来推年付的需要警惕 |
| 价格远低于市场价 | ⚠️ 高 | 月付个位数、买一送三的通常是割韭菜 |
| TG 群禁言/不让讨论 | ⚠️ 中 | 好的服务不怕用户交流 |
| 运营时间 < 6个月 | ⚠️ 中 | 越新的服务不确定性越大 |
| 频繁维护、大面积节点挂掉 | ⚠️ 中 | 技术能力不足的表现 |
| 官网无联系/无客服 | ⚠️ 低 | 跑路了找不到人 |

## 安全信号

- ✅ 支持月付（随时可停）
- ✅ 运营超过一年以上
- ✅ TG 群活跃、用户反馈公开
- ✅ 有退款政策（哪怕只是按比例退）
- ✅ 持续更新订阅配置/协议支持

## 购买建议话术

"先月付试一个月，别买年付。不管什么名字，能用月付解决的问题不要用年付去赌。"

## 验证方法（技术层面）

用户拿到订阅链接后，可以用本技能部署 mihomo 并测速：

1. 部署 mihomo（见 SKILL.md 主流程）
2. 测试连通性：
   ```bash
   curl -s -x http://127.0.0.1:7890 -o /dev/null -w "%{http_code}" https://www.google.com
   ```
3. 测延迟：
   ```bash
   time curl -s -x http://127.0.0.1:7890 -o /dev/null https://www.google.com
   ```
4. 如果用户需要多节点切换延迟对比，可以编写简单的测速脚本
